# Session, token, and cookie handling

## Browser auth cookie
- The browser session cookie name is configurable through `AUTH_COOKIE_NAME`; the default is `fastapiusersauth`. The frontend reads the same value through `SERVER_SIDE_ONLY__AUTH_COOKIE_NAME`. 【F:backend/onyx/configs/constants.py†L34-L43】【F:web/src/lib/constants.ts†L33-L39】
- The fastapi-users cookie transport sets `cookie_max_age` from `SESSION_EXPIRE_TIME_SECONDS` and marks the cookie secure when `WEB_DOMAIN` starts with `https`. 【F:backend/onyx/auth/users.py†L1296-L1300】
- The frontend server-side helpers forward cookies to `/me` and inject a debug auth cookie only in development when `DEBUG_AUTH_COOKIE` is set. 【F:web/src/lib/userSS.ts†L156-L200】

## Session refresh / invalidation
- The Redis auth backend stores a random token in Redis under a key prefix, writes the associated user and tenant ID, refreshes the TTL on refresh, and deletes the token on logout. 【F:backend/onyx/auth/users.py†L1319-L1399】
- The Postgres auth backend updates the access-token expiration in the database on refresh. 【F:backend/onyx/auth/users.py†L1401-L1430】
- The JWT auth backend issues a self-contained JWT with `sub`, `aud`, `iat`, and `exp`; `destroy_token()` is a no-op because JWTs cannot be server-invalidated in that strategy. 【F:backend/onyx/auth/users.py†L1433-L1488】
- The frontend refresh route calls `/auth/refresh` for BASIC/CLOUD/OIDC/GOOGLE_OAUTH deployments. 【F:backend/onyx/main.py†L675-L686】

## OAuth state / PKCE / CSRF cookies
- OAuth state tokens use the `fastapi-users:oauth-state` audience, and the cookie names include `fastapiusersoauthcsrf` plus `fastapiusersoauthpkce_<hash>`. 【F:backend/onyx/auth/users.py†L2112-L2150】
- The OAuth router sets the CSRF cookie and, when PKCE is enabled, the PKCE verifier cookie before redirecting to the provider. 【F:backend/onyx/auth/users.py†L2213-L2312】

## OAuth redirect captcha cookie
- The captcha pre-check for Google OAuth sets a signed `HttpOnly` cookie via `/auth/captcha/oauth-verify`, then `CaptchaCookieMiddleware` validates and clears it after the callback. 【F:backend/onyx/server/auth/captcha_api.py†L60-L123】
- The frontend sign-in button performs the captcha verification call before sending the browser to the authorize URL. 【F:web/src/app/auth/login/SignInButton.tsx†L66-L145】

## PAT / API key handling
- PATs are generated with the `onyx_pat_` prefix, hashed before storage, displayed in masked form, and revoked by expiring the stored record. 【F:backend/onyx/auth/constants.py†L8-L15】【F:backend/onyx/auth/pat.py†L1-L43】【F:backend/onyx/db/pat.py†L110-L200】
- API keys are generated with the `on_` prefix, hashed before storage, displayed in masked form, and regenerated by replacing the stored hash/display values. 【F:backend/onyx/auth/constants.py†L3-L15】【F:backend/onyx/auth/api_key.py†L1-L32】【F:backend/onyx/db/api_key.py†L79-L137】【F:backend/onyx/db/api_key.py†L199-L228】
- The request parser accepts both `Authorization` and `X-Onyx-Authorization` headers for bearer tokens. 【F:backend/onyx/auth/utils.py†L16-L55】

## Client-ready note
- Browser sessions are cookie-based; non-browser access uses bearer tokens. PATs and API keys are stored only as hashes, while OAuth state/CSRF/PKCE and captcha cookies are short-lived, server-validated artifacts. 【F:backend/onyx/auth/users.py†L1296-L1300】【F:backend/onyx/auth/users.py†L2112-L2312】【F:backend/onyx/db/pat.py†L110-L200】【F:backend/onyx/db/api_key.py†L79-L137】
